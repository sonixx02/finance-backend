# Architecture Notes

This v1 keeps the original backend thinking intact while intentionally reducing the public API surface.

## Purpose

The goal is to show clear backend design, not maximum endpoint count. The project keeps the stronger architecture and schema choices that demonstrate system thinking, while trimming the exposed features to the parts that are easiest to explain and test confidently.

## What The Current V1 Exposes

- `POST /auth/login`
- records CRUD under `/records`
- `GET /records/lookups` for Swagger-driven testing
- `GET /dashboard/summary`
- `GET /health`
- `GET /docs`

## What Stays In The Design

### Layered backend structure

- API layer: Express routes and controllers.
- Service layer: orchestration, transactions, and business rules.
- Repository layer: raw PostgreSQL queries.
- Middleware layer: authentication, authorization, validation, and error handling.
- Persistence layer: normalized PostgreSQL schema plus indexes and constraints.

### Dynamic access control

- Roles and permissions stay data-driven through `roles`, `permissions`, `role_permissions`, and `user_roles`.
- `requirePermission.js` continues to authorize requests using permission codes rather than hardcoded role checks.
- JWTs remain short-lived and include token version data so permission changes can invalidate older sessions safely.

### Finance data modeling

- Record types and categories remain stored as data rather than code constants.
- Financial amounts use `NUMERIC(14,2)` to avoid JavaScript floating-point surprises.
- Record type and category compatibility is checked in the service layer and enforced again by the composite foreign key in PostgreSQL.
- Soft delete remains on financial records so history is preserved without physically removing rows.

### Audit support

- `audit_logs` stays in the schema and write flows still record create, update, and delete activity.
- This keeps the system extensible for future admin and compliance-oriented features even though the public API is now smaller.

## What Was Intentionally Removed From The Submission Surface

- Public `/users` management endpoints.
- Dashboard category totals, trends, and recent activity endpoints.
- A public raw OpenAPI endpoint.

These removals reduce reviewer-facing complexity without flattening the underlying system design.

## Why `GET /records/lookups` Was Added

Category public IDs are generated by PostgreSQL, so hardcoding them in docs would be fragile. The lookups endpoint gives reviewers a clean way to fetch valid record types and category IDs before using Swagger to create or update records.

## Future Expansion Without Redesign

Because the richer schema stays in place, the backend can later support:

- admin-facing user creation and role management
- additional analytics endpoints
- richer audit inspection workflows
- more roles and permission combinations
- more finance record types and categories
